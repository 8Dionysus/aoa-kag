# KAG Source Home Preflight

`kag/` is the source-home preflight for KAG-owned local-subtree protocol work in
`aoa-kag`.

It exists so the repository has one obvious home for KAG source-home law before
any source repository starts growing a local `/kag` organ by template copying.

This is about the AoA source-home pattern. It is not evidence that sibling
home directories are named `kag/`.

## Operating Card

| Field | Route |
| --- | --- |
| role | source-home preflight for KAG local-subtree protocol law |
| input | KAG source-home placement pressure, local-subtree protocol pressure, generated-pack posture, and sibling owner-boundary questions |
| output | checked source-home manifest, local-subtree protocol, rollout stop-line, and stronger-owner route |
| owner | `kag/AGENTS.md`, this README, `source_home.manifest.json`, `LOCAL_SUBTREE_PROTOCOL.md`, and decision records |
| next route | `docs/decisions/`, `mechanics/`, `manifests/`, `generated/`, `schemas/`, `examples/`, or the stronger source owner named by the protocol |
| validation | `python scripts/validate_nested_agents.py`, `python -m unittest tests.test_kag_home tests.test_nested_agents_docs`, and broader source-fast when route-facing |

## Why This Home Exists

The current repository already owns KAG substrate docs, manifests, generated
read models, schemas, examples, validators, and mechanics route homes.

The missing surface is different: a KAG source-home route where `aoa-kag` can
define what a future repo-local `/kag` organ means before OS Abyss grows a
forest of local indexes, nodes, edges, projections, and graph views.

This home is that preflight source home.

It is not a payload warehouse.

## Source-Home Pattern Evidence

The sibling pattern is not "every repo has `kag/`." The pattern is that an
organ may have a named source home for its own source objects or posture, with
local route cards, a manifest when formalized, validators, and stop-lines.

| Repository | Home | Pattern carried into `aoa-kag` |
| --- | --- | --- |
| `Tree-of-Sophia` | `ToS/` | broad tree-shaped authored home with branch owners and stronger-owner routes |
| `aoa-agents` | `agents/` | source objects, generated readers, and stronger-owner stop-lines stay distinct |
| `aoa-playbooks` | `playbooks/` | strict canonical source-object paths, no flat aliases |
| `aoa-routing` | `routing/` | local source behavior stays under the home; shared mechanics stay under `mechanics/` |
| `aoa-sdk` | `sdk/` | posture and public promises live in the home while implementation remains elsewhere |
| `aoa-skills` | `skills/` | source bundles differ from generated install/export surfaces |
| `aoa-techniques` | `techniques/` | authored bundles own meaning; mechanics and generated readers stay separate |
| `aoa-memo` | `memo/` | corpus home differs from local memo ports in other repositories |
| `aoa-evals` | `evals/` | proof bundles keep claim meaning local while shared proof infrastructure routes to mechanics |
| `Agents-of-Abyss` | `docs/organ-contract/` | center law says organs need identity, owner boundary, validation, and handoff, not identical trees |

## Future Local Subtree Shape

The protocol currently reserves these future surfaces. They are not active
directories in this repository yet:

| Future surface | Role | Activation condition |
| --- | --- | --- |
| `manifest.json` | declares source refs, generation posture, storage posture, and validation lanes for a repo-local KAG organ | schema, example, validator, and pilot owner exist |
| `nodes/` | source-linked portable node records | node schema and source-ref validator exist |
| `edges/` | bounded direct or derived edge records | edge schema, relation policy, and no-inference validator exist |
| `indexes/` | lightweight lookup indexes and inventory read models | index contract forbids embedding/vector/runtime state |
| `projections/` | compact consumer read models derived from local records | builder and generated parity validator exist |
| `receipts/` | generation, validation, freshness, and regrounding receipts | receipt schema and closeout route exist |

Do not create these surfaces until the activation condition is met.

## Stop-Lines

- No sibling `/kag` rollout from this home yet.
- No live graph, vector, embedding, search, or runtime cache state in Git.
- No source-authored meaning transfer into KAG records.
- No proof verdict, rank mutation, memory acceptance, route dispatch, or ToS
  canon claim.
- No empty directory scaffolding for future protocol pieces.
- No movement of current `generated/`, `manifests/`, `schemas/`, `examples/`,
  `scripts/`, `tests/`, or `mechanics/` payloads merely because `kag/` now
  exists.

## Relationship to Current Surfaces

`kag/` defines the source-home preflight for the future local-subtree protocol.

Current generated KAG outputs stay in `generated/`. Current source-authored
controls stay in `manifests/`. Current payload contracts stay in `schemas/` and
`examples/`. Current deterministic builders and validators stay in `scripts/`.
Repeatable operation pressure stays in `mechanics/` until a real part owns a
payload and validation route.

This home gives the future graph forest a source-home law. It does not plant
the forest yet.
